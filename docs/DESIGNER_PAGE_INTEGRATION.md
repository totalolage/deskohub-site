# Designer Page Integration Procedure

This document outlines the step-by-step process for integrating autogenerated pages from design tools into the DeskohHub codebase, following our **feature-based architecture**.

## Prerequisites

- Access to the exported design files
- Understanding of the feature-based architecture (see PROJECT_STRUCTURE.md)
- Familiarity with our component organization

## Integration Process

### 1. Initial Assessment

When you receive autogenerated files from designers:

1. **Identify imported files using git**
   - Run `git status` to see all untracked files
   - These untracked files are what was imported from designers
   - Typically includes a main page component and several section components

2. **Review the exported files**
   - Note the file structure from git status
   - Identify all components used
   - Check for any assets (images, fonts, etc.)
   - Review the tailwind classes used

2. **Identify the target feature**
   - Determine which business feature this page belongs to
   - If it's a new feature, plan the feature structure
   - If it enhances an existing feature, identify where it fits

3. **Identify conflicts**
   - Headers/footers that duplicate existing ones (use `/features/navigation`)
   - Navigation components (use existing from `/features/navigation`)
   - Any global layout elements
   - Components that already exist in other features

### 2. Component Analysis

1. **Map exported components to existing ones**
   - Check if exported components match existing UI components in `/components/ui/`
   - List components that need to be:
     - Replaced with existing ones
     - Merged into existing ones
     - Created as new components

2. **Check for design system compliance**
   - Verify color usage matches your theme
   - Check spacing and sizing tokens
   - Ensure responsive breakpoints align

### 3. File Organization (Feature-Based)

1. **Determine proper locations based on features**
   - Pages go in `/app/[locale]/[page-name]/page.tsx`
   - Feature components in `/features/[feature-name]/components/`
   - Feature-specific hooks in `/features/[feature-name]/hooks/`
   - Truly shared components in `/shared/components/`
   - UI primitives in `/components/ui/`
   - Assets in `/assets/images/`

2. **Plan the feature structure**
   ```
   features/
   └── [feature-name]/
       ├── components/
       │   ├── FeatureSection.tsx    # Renamed from generic names
       │   └── FeatureComponent.tsx  # e.g., HeroSection → TrainingHero
       ├── hooks/
       │   └── useFeatureLogic.ts
       ├── schemas/
       │   └── featureValidation.ts
       └── index.ts  # Public exports
   
   app/
   └── [locale]/
       └── [new-page]/
           └── page.tsx  # Imports from feature
   ```
   
   **Component Naming Convention:**
   - Prefix components with the feature name for clarity
   - `hero-section.tsx` → `TrainingHero.tsx`
   - `features-section.tsx` → `TrainingFeatures.tsx`
   - `cta-section.tsx` → `TrainingCTA.tsx`

### 4. Code Integration Steps

1. **Remove duplicate layout elements**
   - Remove any common componens such as headers and footers if they exist
   - Keep only the unique page content

2. **Update imports and paths**
   - Replace relative imports with project-standard imports
   - Use the `@/` alias for imports
   - Import existing UI components instead of duplicated ones

3. **Adapt to project conventions**
   ```tsx
   // Replace exported component style:
   export default function PageName() { }
   
   // With project style:
   const PageName = () => {
     return (
       // content
     );
   };
   
   export default PageName;
   ```

4. **Integrate with existing systems**
   - Add i18n support using paraglide
   - Use existing hooks (useLocale, etc.)
   - Apply the cn() utility for className merging
   - Ensure proper TypeScript types
   - Import navigation components from `/features/navigation` instead of using duplicates
   
   **CRITICAL: Set locale in page components**
   - Every page in `/app/[locale]/` MUST call `setLocale()` to ensure i18n works correctly
   - Import both `setLocale` and the route types:
     ```tsx
     import { m, setLocale } from "@/i18n";
     import type { RouteProps_locale } from "../route";
     
     export default async function YourPage({ params }: RouteProps_locale) {
       setLocale((await params).locale);
       
       return (
         // Your page content
       );
     }
     ```
   - Without this, translations will not work and may show default language

5. **Replace hardcoded content**
   - Move text to translation files
   - Replace hardcoded URLs with constants
   - Use environment variables where appropriate

### 5. i18n Integration with Paraglide

**IMPORTANT: Understanding Paraglide Message Keys**

1. **File naming vs. export naming**
   - Paraglide generates files with underscores: `training_hero_title.js`
   - But exports them with dot notation: `export { training_hero_title as "training.hero.title" }`
   - Always check the export statement in the generated file to get the correct key

2. **Message key patterns**
   - Use dot notation when accessing messages: `m["training.hero.title"]()`
   - NOT underscore notation: ~~`m.training_hero_title()`~~
   - Some keys may have numbers appended in filenames (e.g., `training_packages_halfday_title1.js`)

3. **Translation file structure**
   ```json
   {
     "training": {
       "pageTitle": "Training Room - Deskohub",
       "hero": {
         "title": "Team Building",
         "subtitle": "in the training room"
       },
       "packages": {
         "halfDay": {  // Use camelCase for multi-word keys
           "title": "Half-Day Rental"
         }
       }
     }
   }
   ```

4. **Common i18n pitfalls**
   - After adding translations, wait for paraglide to compile (watch the dev server logs)
   - Check `i18n/paraglide/messages/_index.js` to verify your messages are exported
   - Use `grep -E "export {" i18n/paraglide/messages/your_message_*.js` to find correct key names

### 6. Component Standardization

1. **For each component:**
   - Ensure it uses the project's TypeScript patterns
   - Add proper prop interfaces
   - Use CVA for variant styling if applicable
   - Follow the existing component structure

2. **Example transformation:**
   ```tsx
   // From exported:
   export default function HeroSection() {
     return (
       <div className={`bg-blue-500 p-4 ${isActive ? 'border-2' : ''}`}>
         <h1>Welcome to our training room</h1>
       </div>
     );
   }
   
   // To integrated:
   import { cn } from "@/shared/utils";
   import { m } from "@/i18n";
   
   export const TrainingHero = () => {
     return (
       <div className={cn(
         "bg-primary p-4",
         isActive && "border-2",
         className
       )}>
         <h1>{m["training.hero.title"]()}</h1>
       </div>
     );
   };
   ```
   
   **IMPORTANT: Always use `cn()` for className composition**
   - Never use template strings: ~~`className={`text-${size}`}`~~
   - Always use cn(): `className={cn("text-base", size === "large" && "text-lg")}`
   - This ensures proper class merging and avoids Tailwind purging issues

### 7. Asset Management

1. **Images and media**
   - Move to `/assets/images/`
   - Use proper naming conventions
   - Update import paths

2. **Icons**
   - Check if icons exist in lucide-react
   - Replace with existing icon library when possible

### 8. Styling Integration

1. **Tailwind classes**
   - Ensure classes use theme tokens (e.g., `bg-primary` not `bg-blue-500`)
   - Check responsive utilities match project patterns
   - Verify dark mode support if applicable

2. **Custom styles**
   - Move any custom CSS to component-level styles
   - Use CSS modules only if absolutely necessary (following existing patterns), prefer tailwind wherever possible

3. **className Best Practices**
   ```tsx
   // ❌ NEVER use template strings for dynamic classes
   <div className={`text-${size} ${isActive ? "bg-blue" : "bg-gray"}`}>
   
   // ✅ ALWAYS use cn() for className composition
   import { cn } from "@/shared/utils";
   
   <div className={cn(
     "text-base",
     size === "large" && "text-lg",
     isActive ? "bg-blue" : "bg-gray"
   )}>
   
   // ✅ With CSS Modules
   <div className={cn(styles.myClass, "additional-classes")}>
   
   // ✅ Complex conditions
   <Button
     className={cn(
       "w-full",
       variant === "primary" && "bg-primary hover:bg-primary/90",
       variant === "secondary" && "bg-secondary hover:bg-secondary/90",
       disabled && "opacity-50 cursor-not-allowed"
     )}
   >
   ```
   
   **Why use cn():**
   - Prevents Tailwind purging issues with dynamic classes
   - Properly merges conflicting classes
   - Cleaner, more readable code
   - Type-safe when used with TypeScript

### 9. Testing and Validation

**Server Logs Monitoring**
- Keep an eye on the dev server logs when developing
- If using a piped log (e.g., `/tmp/deskohub.log`), use `tail -f` to monitor
- Look for import errors related to i18n - these indicate incorrect message keys
- Example: `TypeError: m.training_hero_title is not a function` means use `m["training.hero.title"]()`

1. **Type and lint checking**
   ```bash
   bun run check
   ```

2. **Build verification**
   ```bash
   bun run build
   ```

3. **Visual testing**
   - Check all breakpoints
   - Verify i18n works correctly
   - Test interactive elements

4. **Cross-browser testing**
   - Test in major browsers
   - Check mobile responsiveness

### 10. Cleanup

1. **Remove unused code**
   - Delete any components that were fully replaced
   - Remove unused imports
   - Clean up any generated comments

2. **Code formatting**
   - Run formatter to ensure consistency
   - Fix any linting issues

## Common Patterns to Watch For

### Headers and Footers
Always remove exported headers/footers and use the existing ones from:
- `/features/navigation` (Header and Footer components)

### Navigation
Use the existing navigation system from `/features/navigation` rather than any exported navigation.

### Form Components
Map to existing form components in `/components/ui/form.tsx` and related files.

### Buttons and Links
Use the existing Button component from `/components/ui/button` with appropriate variants.

### Layout Containers
Use consistent max-width containers and padding as seen in existing pages.

## Integration Example: Pricing Page

Let's walk through integrating a pricing page export:

### 0. Identifying Imported Files
```bash
$ git status
Untracked files:
  (use "git add <file>..." to include in what will be committed)
    pricing-page.tsx
    components/pricing-hero.tsx
    components/pricing-card.tsx
    components/pricing-faq.tsx
    components/header.tsx
    components/footer.tsx
    assets/pricing-bg.jpg
```

### 1. Exported Structure (from designer)
```
pricing-export/
├── PricingPage.tsx
├── PricingHero.tsx
├── PricingCard.tsx
├── PricingFAQ.tsx
├── Header.tsx        # Duplicate
├── Footer.tsx        # Duplicate
└── assets/
    └── pricing-bg.jpg
```

### 2. Integrated Structure (feature-based)
```
features/
└── pricing/            # New feature
    ├── components/
    │   ├── PricingHero.tsx
    │   ├── PricingCard.tsx
    │   └── PricingFAQ.tsx
    ├── types/
    │   └── pricing.types.ts
    └── index.ts

app/
└── [locale]/
    └── pricing/
        └── page.tsx    # Uses components from feature

assets/
└── images/
    └── pricing-bg.jpg
```

### 3. Feature Index File
```typescript
// features/pricing/index.ts
export { PricingHero } from "./components/PricingHero";
export { PricingCard } from "./components/PricingCard";
export { PricingFAQ } from "./components/PricingFAQ";
export type { PricingTier, FAQItem } from "./types/pricing.types";
```

### 4. Page Implementation
```typescript
// app/[locale]/pricing/page.tsx
import { PricingHero, PricingCard, PricingFAQ } from "@/features/pricing";
import { m, setLocale } from "@/i18n";
import type { RouteProps_locale } from "../route";
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: m["pricing.pageTitle"](),
  description: m["pricing.pageDescription"](),
};

export default async function PricingPage({ params }: RouteProps_locale) {
  // CRITICAL: Always set the locale at the beginning of page components
  setLocale((await params).locale);

  return (
    <>
      <PricingHero />
      <div className="grid grid-cols-1 md:grid-cols-3 gap-8">
        <PricingCard tier="basic" />
        <PricingCard tier="pro" featured />
        <PricingCard tier="enterprise" />
      </div>
      <PricingFAQ />
    </>
  );
}
```

## Troubleshooting Guide

### Locale Not Working

**Problem**: Page shows wrong language or default language despite being on correct URL
**Solution**: Add `setLocale()` call at the beginning of your page component:
```tsx
export default async function YourPage({ params }: RouteProps_locale) {
  setLocale((await params).locale); // Add this line!
  // ... rest of your page
}
```

### i18n Message Errors

**Problem**: `TypeError: m.message_key is not a function`
**Solution**: Use bracket notation with dots: `m["message.key"]()`

**Problem**: Message not found after adding translations
**Solution**: 
1. Check if paraglide compiled (watch server logs)
2. Verify message exists in `i18n/paraglide/messages/_index.js`
3. Check the exact export name in the generated file

### Component Import Errors

**Problem**: Components from designer import have default exports
**Solution**: Convert to named exports with arrow functions:
```tsx
// Change this:
export default function Component() { }

// To this:
export const Component = () => { };
```
