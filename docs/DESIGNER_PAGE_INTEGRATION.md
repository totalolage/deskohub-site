# Designer Page Integration Procedure

This document outlines the step-by-step process for integrating autogenerated pages from design tools into the DeskohHub codebase, following our **feature-based architecture**.

## Prerequisites

- Access to the exported design files
- Understanding of the feature-based architecture (see PROJECT_STRUCTURE.md)
- Familiarity with our component organization

## Integration Process

### 1. Initial Assessment

When you receive autogenerated files from designers:

1. **Review the exported files**
   - Note the file structure
   - Identify all components used
   - Check for any assets (images, fonts, etc.)
   - Review the tailwind classes used

2. **Identify the target feature**
   - Determine which business feature this page belongs to
   - If it's a new feature, plan the feature structure
   - If it enhances an existing feature, identify where it fits

3. **Identify conflicts**
   - Headers/footers that duplicate existing ones (use `/features/navigation`)
   - Navigation components (use existing from `/features/navigation`)
   - Any global layout elements
   - Components that already exist in other features

### 2. Component Analysis

1. **Map exported components to existing ones**
   - Check if exported components match existing UI components in `/components/ui/`
   - List components that need to be:
     - Replaced with existing ones
     - Merged into existing ones
     - Created as new components

2. **Check for design system compliance**
   - Verify color usage matches your theme
   - Check spacing and sizing tokens
   - Ensure responsive breakpoints align

### 3. File Organization (Feature-Based)

1. **Determine proper locations based on features**
   - Pages go in `/app/[locale]/[page-name]/page.tsx`
   - Feature components in `/features/[feature-name]/components/`
   - Feature-specific hooks in `/features/[feature-name]/hooks/`
   - Truly shared components in `/shared/components/`
   - UI primitives in `/components/ui/`
   - Assets in `/assets/images/`

2. **Plan the feature structure**
   ```
   features/
   └── [feature-name]/
       ├── components/
       │   ├── FeatureSection.tsx
       │   └── FeatureComponent.tsx
       ├── hooks/
       │   └── useFeatureLogic.ts
       ├── schemas/
       │   └── featureValidation.ts
       └── index.ts  # Public exports
   
   app/
   └── [locale]/
       └── [new-page]/
           └── page.tsx  # Imports from feature
   ```

### 4. Code Integration Steps

1. **Remove duplicate layout elements**
   - Remove any common componens such as headers and footers if they exist
   - Keep only the unique page content

2. **Update imports and paths**
   - Replace relative imports with project-standard imports
   - Use the `@/` alias for imports
   - Import existing UI components instead of duplicated ones

3. **Adapt to project conventions**
   ```tsx
   // Replace exported component style:
   export default function PageName() { }
   
   // With project style:
   const PageName = () => {
     return (
       // content
     );
   };
   
   export default PageName;
   ```

4. **Integrate with existing systems**
   - Add i18n support using paraglide
   - Use existing hooks (useLocale, etc.)
   - Apply the cn() utility for className merging
   - Ensure proper TypeScript types

5. **Replace hardcoded content**
   - Move text to translation files
   - Replace hardcoded URLs with constants
   - Use environment variables where appropriate

### 5. Component Standardization

1. **For each component:**
   - Ensure it uses the project's TypeScript patterns
   - Add proper prop interfaces
   - Use CVA for variant styling if applicable
   - Follow the existing component structure

2. **Example transformation:**
   ```tsx
   // From exported:
   <div className="bg-blue-500 p-4">
     <h1>Welcome</h1>
   </div>
   
   // To integrated:
   import { cn } from "@/shared/utils";
   import { m } from "@/i18n";
   
   <div className={cn("bg-primary p-4", className)}>
     <h1>{m["welcome.title"]()}</h1>
   </div>
   ```

### 6. Asset Management

1. **Images and media**
   - Move to `/assets/images/`
   - Use proper naming conventions
   - Update import paths

2. **Icons**
   - Check if icons exist in lucide-react
   - Replace with existing icon library when possible

### 7. Styling Integration

1. **Tailwind classes**
   - Ensure classes use theme tokens (e.g., `bg-primary` not `bg-blue-500`)
   - Check responsive utilities match project patterns
   - Verify dark mode support if applicable

2. **Custom styles**
   - Move any custom CSS to component-level styles
   - Use CSS modules only if absolutely necessary (following existing patterns), prefer tailwind wherever project

### 8. Testing and Validation

1. **Type and lint checking**
   ```bash
   bun run check
   ```

2. **Build verification**
   ```bash
   bun run build
   ```

3. **Visual testing**
   - Check all breakpoints
   - Verify i18n works correctly
   - Test interactive elements

4. **Cross-browser testing**
   - Test in major browsers
   - Check mobile responsiveness

### 9. Cleanup

1. **Remove unused code**
   - Delete any components that were fully replaced
   - Remove unused imports
   - Clean up any generated comments

2. **Code formatting**
   - Run formatter to ensure consistency
   - Fix any linting issues

## Common Patterns to Watch For

### Headers and Footers
Always remove exported headers/footers and use the existing ones from:
- `/features/navigation` (Header and Footer components)

### Navigation
Use the existing navigation system from `/features/navigation` rather than any exported navigation.

### Form Components
Map to existing form components in `/components/ui/form.tsx` and related files.

### Buttons and Links
Use the existing Button component from `/components/ui/button` with appropriate variants.

### Layout Containers
Use consistent max-width containers and padding as seen in existing pages.

## Integration Example: Pricing Page

Let's walk through integrating a pricing page export:

### 1. Exported Structure (from designer)
```
pricing-export/
├── PricingPage.tsx
├── PricingHero.tsx
├── PricingCard.tsx
├── PricingFAQ.tsx
├── Header.tsx        # Duplicate
├── Footer.tsx        # Duplicate
└── assets/
    └── pricing-bg.jpg
```

### 2. Integrated Structure (feature-based)
```
features/
└── pricing/            # New feature
    ├── components/
    │   ├── PricingHero.tsx
    │   ├── PricingCard.tsx
    │   └── PricingFAQ.tsx
    ├── types/
    │   └── pricing.types.ts
    └── index.ts

app/
└── [locale]/
    └── pricing/
        └── page.tsx    # Uses components from feature

assets/
└── images/
    └── pricing-bg.jpg
```

### 3. Feature Index File
```typescript
// features/pricing/index.ts
export { PricingHero } from "./components/PricingHero";
export { PricingCard } from "./components/PricingCard";
export { PricingFAQ } from "./components/PricingFAQ";
export type { PricingTier, FAQItem } from "./types/pricing.types";
```

### 4. Page Implementation
```typescript
// app/[locale]/pricing/page.tsx
import { PricingHero, PricingCard, PricingFAQ } from "@/features/pricing";
import { setLocale } from "@/i18n";
import type { RouteProps_locale } from "../route";

export default async function PricingPage({ params }: RouteProps_locale) {
  setLocale((await params).locale, { reload: false });

  return (
    <>
      <PricingHero />
      <div className="grid grid-cols-1 md:grid-cols-3 gap-8">
        <PricingCard tier="basic" />
        <PricingCard tier="pro" featured />
        <PricingCard tier="enterprise" />
      </div>
      <PricingFAQ />
    </>
  );
}
```

mkdir -p features/pricing/components features/pricing/types

# After integration
bun run check
bun run build

# Test in development
bun run dev

# Commit when ready
git add .
git commit -m "feat: add pricing feature from design export"
